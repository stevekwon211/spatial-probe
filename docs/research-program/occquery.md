# Stage 2 — OccQuery v1: an externally-validated denotation-correctness benchmark for occupancy-native static-geometry predicates, with an independent (non-circular) LiDAR-derived oracle and a pre-registered ≥20-F1 expressivity gap over box-only retrieval. This is the second iteration of the OccQuery experiment (program-position 1 in the six-paper PLAN), NOT the dynfield program-stage. Stage 1/v0 produced (a) a tested pure-CPU core (raycast DDA, C-space reachable_free_field, three free-space predicates, Occ3D-nuScenes adapter, query DSL, retrieval; 72 tests green on numpy+scipy+pyyaml, zero torch/CUDA), (b) the oracle-free H1 expressivity witness as 3 passing tests (tests/test_expressivity.py), and (c) a v0→v1 hardening pass on real Occ3D-mini that exposed and root-cause-fixed five false-positive modes (the C-space rewrite in src/probe/predicates/reachable.py) and re-scoped OccQuery to STATIC MEASUREMENT (danger → dynfield). What Stage 1 does NOT yet have, and what Stage 2 delivers, is the single load-bearing publishable artifact: a real H3 denotation P/R/F1 against an INDEPENDENT oracle (the current docs/h3-real-data-findings.md pass is self-flagged circular "visual-agreement", L6-10/L87-89 — the GT is a human call on the same dense-GT slice the predicate reads, so it is not external evidence and is explicitly not a scientific result).

## Motivation
Motivation. AV scenario mining today is box-native: RefAV (arXiv:2505.20981) translates natural-language queries into 33 atomic functions over tracked-object cuboids + HD-map polygons (docs/expressivity-vs-refav.md L9-23), and nuPlan/Waymo mine over object tracks. A whole class of safety-relevant geometry lives in the SPACE BETWEEN boxes — the physical gap to an unboxed wall/debris/curb/protruding-load, the free width remaining in a corridor, whether the ego's swept footprint stays in reachable free space — and a box+map observation cannot denote it: an obstacle with no uuid is invisible to near_objects(), and in_drivable_area() reads a static map polygon, not the scene's current occupancy (docs/expressivity-vs-refav.md L34-47). Dense occupancy fields (Occ3D-nuScenes) now make these quantities computable from a released public substrate.\n\nThe precise gap (three layers, ordered by how hard each is to dispute, PLAN L117-122). (1) An oracle-free EXPRESSIVITY separation: the queries are inexpressible in RefAV's released function set — already shown as a syntactic separation (grep returns 0 free-space primitives in 33 functions, verified 2026-06-20, docs/expressivity-vs-refav.md L30-32) plus a non-identifiability witness (tests/test_expressivity.py: two scenes with identical box observables differing only by an unboxed abeam wall; the box backend MUST return the same answer, the occupancy predicate distinguishes them, L57-71). (2) A DENOTATION-CORRECTNESS benchmark: do the predicates compute the geometric quantity CORRECTLY on real data, measured against an oracle built from a DIFFERENT data source than the predicate reads? This does not exist yet and is the Stage-2 deliverable. (3) A third-party leaderboard anchor (H2, RefAV HOTA-Temporal on EvalAI) — deferred from Stage 2 because of a substrate mismatch (RefAV is Argoverse 2, our occupancy substrate is Occ3D-nuScenes; docs/m2-adapter-contract.md L81-88 flags this as "the single biggest open question") and because it adds no CPU-feasible signal on a Mac.\n\nWhy now, why this is the cheap honest move. v0 runs on GT occupancy with pure-CPU geometric ops (numpy + scipy; pyproject.toml dependencies are numpy>=1.26, pyyaml>=6.0; data extra = nuscenes-devkit; zero torch/CUDA — confirmed by grep over src/). The independent oracle (raw accumulated multi-sweep LiDAR → exact EDT clearance, the Voxblox/FIESTA recompute pattern) is also pure-CPU. So the entire externally-validated result is reachable on a laptop with no GPU. The frontier lesson the lab already embodies (synthesis §3, §7) is that the strongest scientific move is to reform MEASUREMENT — release a construction so a wrong claim is structurally hard to make — and OccQuery's H1 witness + the to-be-built independent oracle are exactly that. This also pays for ~3 of the 6 program experiments' shared substrate (raycast + Occ3D adapter + visibility-mask handling; PLAN L332-334).

## Central hypothesis
Deterministic occupancy-native physical predicates (lateral_clearance, free_along_ego_path, min_free_width_along_path) executed over a dense occupancy field measure box-blind static free-space geometry on real logged driving scenes both (i) EXPRESSIVELY — denoting queries that RefAV's released box-only function set cannot express at all — and (ii) DENOTATION-CORRECTLY — agreeing with an INDEPENDENT geometric oracle (raw accumulated-LiDAR free-space, reconstructed apart from the Occ3D field the predicates read), and beating the best box-only RefAV approximation of the same query by a LARGE RELATIVE margin (≥20 absolute F1), with that separation STABLE across the three UnknownPolicy rules (free/occupied/ignored). The load-bearing claim is the RELATIVE expressivity/denotation gap over box-only, not any absolute F1 cutoff. Scope guard (PLAN L124-137; non-negotiable): the hypothesis is about MEASUREMENT CORRECTNESS of static geometry, never about whether a measured situation is dangerous — a 0.8 m free-width beside a lead car is a CORRECT measurement, and "it's a lead car not a wall" is a dynfield verdict, not an OccQuery failure.

## Sub-hypotheses
- H1 (expressivity, oracle-free, the headline — HARDEN, do not re-derive). The N curated occupancy queries are inexpressible in RefAV's released 33-function set, and a box+map observation is provably insufficient to denote them. Strongest form = the non-identifiability witness (tests/test_expressivity.py): scenes with identical tracked-box observables differing only by unboxed occupancy force any box-only function to return identical answers while the occupancy predicate distinguishes them. Stage-2 work: (a) generalize the witness from one abeam-wall pair to ALL three predicate families (corridor-narrows, free-path-blocked, lateral-clearance) so each query in the released set has a paired witness; (b) tighten the honesty bound — the claim is against the RefAV function set + the tracked-box observable set, NOT a proof that no conceivable box-language could approximate it with extra engineering (docs/expressivity-vs-refav.md L50-61; tests/test_expressivity.py L11-14). Oracle dependence: NONE. This is the quasi-deductive leg (synthesis §6).
- H3a (denotation correctness vs an INDEPENDENT oracle — the core Stage-2 deliverable). On a labeled real subset, each predicate's continuous output (clearance metres, free-width metres) and the derived per-query boolean match an oracle CONSTRUCTED FROM A DIFFERENT DATA SOURCE than the predicate reads: raw accumulated multi-sweep LiDAR re-voxelized and distance-transformed (exact EDT), reconstructed apart from the Occ3D semantics field. Report continuous MAE + tolerance-accuracy ([75%,125%]) AND boolean denotation P/R/F1, each with per-scene bootstrap 95% CIs. This is what makes the result a scientific result rather than the current circular visual-agreement pass (docs/h3-real-data-findings.md L6-10, L87-89). Oracle dependence: released oracle (construction code + held-out scene IDs published; benchmark-anchors.md L98-100).
- H3b (the ≥20-F1 relative gap over box-only — the load-bearing comparison). For each occupancy query, build the BEST box-only RefAV approximation (compose near_objects / has_objects_in_relative_direction / in_drivable_area over the SAME scenes' tracked boxes + HD-map), score its boolean denotation against the SAME independent oracle, and show free-space predicates beat it by ≥20 absolute F1. The relative gap is load-bearing; absolute F1 (target ≥0.90) is only a secondary internal-validity check (PLAN L160, L187-190; benchmark-anchors.md L101-102). Falsifies cleanly: if the box-only approximation matches denotation quality, free-space predicates add no retrieval power and the angle fails (PLAN L192).
- H3c (UnknownPolicy 3-rule stability — the geometric-mechanism isolation, MECE ablation). On the dense GT oracle (~0 unknown) the denotation P/R/F1 and the ≥20-F1 gap are STABLE across UnknownPolicy ∈ {FREE, OCCUPIED, IGNORED}; if instability appears on the released field, scope shrinks to dense-LiDAR accumulated free-space and re-tests (PLAN L171-176, L192-195; benchmark-anchors.md L99-110). This is the named main-confounder test (unobserved voxels), not a nuisance — and the UNKNOWN→-1 mapping is what makes it real on actual sensor coverage rather than synthetic (docs/m2-adapter-contract.md L43-51). The per-frame visibility mask (~88% unknown) is deliberately NOT applied to the H3 oracle; it is the conditioning variable for the gt-distrust/vis-calibration stages instead (PLAN L173-177).

## Success criteria (pre-registered)
- PRE-REGISTERED EVALUATION BUDGET (fix before any number is read, per synthesis §2 — the field's biggest gap is no fixed budget; RT-2/OpenVLA-style). Declare up front: N≥20 occupancy queries in the released set (only 4 exist today: experiments/occquery_v0/queries.yaml — 3 occupancy + 1 baseline; PLAN targets ~20); a labeled scene set that CONTAINS POSITIVES (the 10 Occ3D-mini scenes have none, so recall is currently unmeasurable — docs/h3-real-data-findings.md L107-108) — minimum 60 scenes drawn from nuScenes val with a pre-declared positive-containing protocol; exactly 3 UnknownPolicy rules; bootstrap N=1000 resamples for all CIs. All of N, the scene IDs, and the cutoffs are committed to the repo (results/preregistration.md) BEFORE the oracle is run.
- H1 PASS (oracle-free headline). Expressivity separation holds: occupancy ≈ N queries expressible, RefAV ≈ 0 (re-confirmed by grep against the live atomic_functions.py + a paired non-identifiability witness for EACH of the three predicate families, all green in tests/test_expressivity.py). This is necessary but explicitly NOT sufficient — it is the existence/diagnosis lane (synthesis §8).
- H3a PASS. On the labeled val subset, boolean denotation F1 ≥ 0.90 absolute (SECONDARY internal-validity check only) AND continuous clearance MAE reported with its full distribution + tolerance-accuracy in [75%,125%], each with per-scene bootstrap 95% CIs — measured against the INDEPENDENT raw-LiDAR oracle (different data source from the Occ3D field the predicate reads).
- H3b PASS (LOAD-BEARING). Free-space predicates beat the best box-only RefAV approximation of the same query by ≥ 20 absolute F1 on the SAME oracle, reported as a FULL continuous curve (F1-gap vs decision threshold), not a single cutoff — a threshold a reviewer can move is not a result (PLAN L184-185; benchmark-anchors.md L28-31). The relative gap, not the absolute 0.90, is the claim.
- H3c PASS. The ≥20-F1 gap and the denotation F1 are stable (overlapping CIs) across all three UnknownPolicy rules on the dense GT; the per-rule numbers are reported as a 3-row table, never averaged into one (no-cross-protocol-averaging discipline).
- REPRODUCIBILITY GATES. Oracle-construction code + held-out scene IDs released (benchmark-anchors.md L25-27, L98-100); data/VERSIONS.txt pins nuScenes version + Occ3D commit; data/checksums.txt records sha256 of every labels.npz (docs/m2-adapter-contract.md L32-36); H1 ships a copy-paste curl+grep+pytest reproduce block (docs/expressivity-vs-refav.md L64-69). Every claim verified THREE ways — code + data + rendered image — per the no-number-only rule (docs/h3-real-data-findings.md L133-141).
- Publishable-tier per benchmark-anchors.md L95-102 reached (public Occ3D-nuScenes val + released oracle + ≥20-F1 gap + 3-policy stability). 'Strong' tier (H2 HOTA-Temporal ≥50 on EvalAI) is explicitly OUT of Stage 2 (deferred for substrate mismatch + GPU/leaderboard reasons; see scope_out).

## Kill / pivot criteria (pre-registered)
- BOX-ONLY MATCHES (primary kill, PLAN L192; benchmark-anchors.md L107-108). If the best box-only RefAV approximation matches the free-space predicates' denotation quality against the independent oracle — i.e. the ≥20-F1 gap collapses (CIs overlap or gap <20 F1) — then occupancy adds no retrieval power over boxes+map for these queries and the central hypothesis is FALSIFIED. Report the negative as the headline, not a footnote (synthesis §2): 'on this substrate, box+map composition recovers free-space geometry, so the occupancy-native angle does not hold here.'
- UNKNOWN-POLICY FLIP (scope-shrink pivot, PLAN L193-195; benchmark-anchors.md L108-110). If denotation flips across the 3 UnknownPolicy rules EVEN ON the released/dense field (gap or F1 swings beyond CI between rules), the geometry is not robust to unobserved-voxel ambiguity → PIVOT: shrink scope to dense-LiDAR accumulated free-space (multi-sweep) and re-test. An acceptable, honest result, not a failure.
- ORACLE STILL CIRCULAR (validity kill). If the 'independent' oracle cannot be built from a genuinely different data source — e.g. raw-LiDAR re-voxelization collapses back to the Occ3D field (same accumulation/TSDF pipeline) — then H3a is still self-graded and the result stays below Publishable; STOP and report only H1 (oracle-free) as the result, exactly as Stage 1 already does (docs/h3-real-data-findings.md L88-89). Do not upgrade a circular pass to a denotation claim.
- RESOLUTION FLOOR (scope honesty, not a kill — declare it). At 0.4 m Occ3D voxels a sub-voxel gap (e.g. a 0.07 m car parked beside, scene-0757) cannot be told from contact; obstacle voxels fall inside the ego footprint (docs/h3-real-data-findings.md L210-214). Any query whose true clearance is <0.4 m is OUT OF SCOPE for the Occ3D arm and must be excluded from the labeled set or routed to a raw-LiDAR re-voxelization — it is a data-resolution ceiling, not a predicate bug. (Separately fix lateral_clearance's voxel-half guard, which under-reports near-contact: it reported 0.27 m where true nearest was 0.07 m, src/probe/predicates/clearance.py L69.)
- NO POSITIVES (measurability kill). If the assembled val subset cannot be made to contain real positives for a given query family under the pre-declared protocol, recall for that family is unmeasurable and that family is dropped from the quantitative claim (kept only in H1 expressivity), rather than reporting precision-only numbers that imply it works (the v0 error: 'every time it speaks it is wrong', docs/h3-real-data-findings.md L83-84).
- DANGER-RETRIEVAL CREEP (category-error guard, hard rule). If at any point success is being measured by whether retrieved scenes are 'dangerous' / 'real near-misses' rather than whether the MEASUREMENT is geometrically correct, STOP — that is the v0 0/5 category error (a static instrument graded by a dynamic answer key, PLAN L132-135; docs/h3-real-data-findings.md L189-201). Danger is the dynfield experiment.

## Oracle
"INDEPENDENT raw-LiDAR free-space oracle (the core Stage-2 build; pure-CPU). Construction, exact-recompute family (synthesis §3-A, the Voxblox/FIESTA/nvblox pattern): for each labeled keyframe, accumulate the nuScenes LiDAR sweeps over a short window (static scene in global frame; the ego footprint and dynamic boxes handled in object frame), re-voxelize the raw points into an occupied/free/unknown grid at the SAME ego-centric extent as Occ3D but from the SENSOR RETURNS, not the Occ3D semantics field, then compute clearance and free-width by an exact Euclidean distance transform over that independent grid. The oracle's value for a query is this geometric quantity; the boolean GT is the threshold applied to it. This is a DIFFERENT data source from what the predicate reads (the predicate reads Occ3D semantics; the oracle reads raw LiDAR), which is exactly what the current pass lacks.\n\nWHY THIS IS NOT CIRCULAR (the rule the lab already enforces, benchmark-anchors.md L216-225; synthesis §3 trap-1). The current docs/h3-real-data-findings.md pass is circular by its own admission: the GT is a human call on the SAME dense-GT slice the predicate runs on, so high agreement is expected and proves nothing external (L6-10, L87-89). The fix is structural: the oracle must share NEITHER the data source NOR the computation with the predicate. Raw-LiDAR re-voxelization + exact EDT satisfies both — a disagreement between predicate (Occ3D-derived) and oracle (LiDAR-derived) is now genuine external evidence, and an AGREEMENT is informative. Validity check before trusting it: confirm the raw-LiDAR pipeline does NOT reduce to the Occ3D accumulation/TSDF (if Occ3D itself was built by VDBFusion over these same sweeps, accumulate a DISJOINT sweep window or use a different reconstruction so the two are not the same operation — if they cannot be separated, this triggers the 'oracle still circular' kill).\n\nVISIBILITY MASK IS PROTOCOL-DEFINING AND DELIBERATELY HELD OUT (synthesis §3 trap-2; PLAN L170-177). The H3 oracle uses the dense, near-fully-observed field (~0 unknown), so denotation depends on geometry, not on the unknown policy. The per-frame visibility mask (~88% unknown on a single sweep) is NOT applied to the oracle — masking the dense GT makes denotation depend on the policy and produces the kill-shape on real data (PLAN L172-174). That masked observed view is reserved as the conditioning variable for gt-distrust / vis-calibration, NOT used here. The UNKNOWN→-1 encoding (src/probe/raycast.py) is what makes the 3-rule sensitivity (H3c) real on actual sensor coverage (docs/m2-adapter-contract.md L43-51).\n\nRELEASE (benchmark-anchors.md L25-27, L98-100; PLAN L177). Oracle-construction code + the held-out scene IDs are published so anyone can recompute it; self-oracle is never the headline (H1 leads, the oracle backs H3a/H3b). Synthetic scenes (experiments/occquery_v0/synthetic.py, F1=1.00 by construction) remain a SMOKE test of the loop only — never evidence (PLAN L202-204) — and are used in Stage 2 solely to add the five audit failure modes as adversarial regression negatives (docs/h3-real-data-findings.md L108)."

- Predicate-geometry: v0 centerline-scan vs v1 C-space reachable_free_field (src/probe/predicates/reachable.py). - Physical-gap correction: centerline_lateral_distance (center-to-center, raw) vs lateral_clearance (body-surface-to-obstacle-surface). Isolates whether subtracting ego-half + voxel-half matters for denotation (clearance.py L24-74).
- min_cluster_voxels ∈ {1, 2} — single-frame noise robustness (reachable.py L120-121). - horizon ∈ {1.0, 2.0} s for free_along_ego_path / min_free_width_along_path — isolates how far the constant-velocity reach extends the corridor (freepath.py). A geometric (not temporal) knob: it sets the swept extent, not a dynamics claim.
- Resolution sensitivity: Occ3D 0.4 m voxels vs raw-LiDAR re-voxelized finer grid on the SAME scenes — quantifies the sub-voxel ceiling (docs/h3-real-data-findings.md L210-214) and tells which queries are resolution-limited vs predicate-limited. NO factorial interaction ablation is in scope for Stage 2 (no predicted-occupancy arm to cross with); when the v1 GPU arm lands later, cross {UnknownPolicy × occupancy-source} to expose interaction (synthesis §5 warns against the monotone ladder).

## Scope — in
- EXPRESSIVITY (H1): the occupancy queries are inexpressible in RefAV's released box-only function set, and box+map observations are provably insufficient to denote them (witness, per predicate family). Oracle-free.
- DENOTATION CORRECTNESS (H3a): the three free-space predicates (lateral_clearance, free_along_ego_path, min_free_width_along_path) compute their geometric quantity correctly on real Occ3D-nuScenes val scenes, measured against an INDEPENDENT raw-LiDAR oracle.
- RELATIVE GAP (H3b): free-space predicates beat the best box-only RefAV approximation by ≥20 absolute F1 against that same oracle — the load-bearing claim.
- 3-RULE STABILITY (H3c): denotation is stable across UnknownPolicy {free/occupied/ignored} on the dense GT.
- STATIC, SINGLE-FRAME geometry only — free-width, clearance, free-path-blocked-in-some-frame (scope=any). The released oracle-construction code + held-out scene IDs. Pure-CPU on a Mac: numpy + scipy + nuscenes-devkit, no GPU at any step.
- Releasing the benchmark (predicates + oracle + queries + metrics + held-out IDs) as a reproducible artifact others can recompute.

## Scope — out (what it does NOT claim)
- DANGER / risk / near-miss retrieval (HARD out, PLAN L124-137). OccQuery returns a static geometric FACT (free-width=0.8 m, side gap=0.27 m, path blocked=true); whether that is dangerous needs relative motion over time = the dynfield experiment. A 0.8 m gap beside a lead car is a CORRECT measurement, not a false positive. Success is measurement accuracy + expressivity, NEVER danger-retrieval F1.
- TEMPORAL / dynamics claims: blocked→clears transitions, 'it actually pulled away', multi-frame persistence, relative motion. The v0 transition query was demoted to a static scope=any 'free_path_is_blocked' (docs/h3-real-data-findings.md L151-166); the temporal version is dynfield.
- H2 HOTA-Temporal leaderboard placement (deferred from Stage 2). Reasons: (a) substrate mismatch — RefAV/EvalAI is Argoverse 2, the occupancy substrate is Occ3D-nuScenes (docs/m2-adapter-contract.md L81-88); (b) it needs the EvalAI challenge rules confirmed to permit a deterministic-predicate-over-occupancy submission (PLAN L151-154); (c) it adds no CPU-feasible denotation signal. ### Logic (inductive/deductive soundness) — adversarial top-venue review lens — verdict: revise — the deductive H1 leg is sound and genuinely oracle-free, but the load-bearing Stage-2 deliverable (H3a/H3b denotation correctness against an "independent" oracle) rests on a non-independence that the code itself contradicts, and the falsification structure lets every disconfirming outcome escape into a still-publishable H1. As written it cannot establish the thing it claims is the Stage-2 contribution. Not reject-as-unsound because the fixes are concrete and the H1 spine survives; but the central H3 claim must be re-architected before this is a denotation-correctness benchmark rather than a relabeled Stage-1.
- **blocker**: CIRCULAR ORACLE — the 'independent' oracle is not an independent data source; the claim is affirming-the-consequent. The predicate reads Occ3D semantics, which the repo's own adapter documents as 'a multi-sweep accumulated [LiDAR] label' (src/probe/adapters/occ3d.py L48-49). The proposed oracle is 'raw accumulated multi-sweep LiDAR re-voxelized.' Occ3D-nuScenes GT is itself built by accumulating the SAME nuScenes LiDAR sweeps + voxelizing at 0.4 m. So predicate-source and oracle-source share the same physical signal (identical LiDAR returns), the same sensor, the same accumulation operation, and the same voxel grid. That is one data source measured twice, not two. The plan's syllogism 'oracle reads raw LiDAR, predicate reads Occ3D, therefore different sources' is invalid: Occ3D = a function of raw LiDAR, so agreement is expected by construction (the EXACT failure mode the plan says it fixes). High predicate-vs-oracle F1 would confirm 'two voxelizations of the same LiDAR agree,' not 'the predicate measures real geometry.'
  - fix: Demote the LiDAR-re-voxelization oracle from 'independent' to 'same-modality reconstruction control' and state plainly it bounds re-voxelization/voxel-size error, NOT external denotation correctness. For a genuinely independent oracle, change MODALITY, not the reconstruction operation: derive clearance/free-width from a source whose errors are uncorrelated with LiDAR voxelization — e.g. stereo/mono-depth geometry, HD-map curb/barrier polygons + tracked-box hulls for the boxable subset, or hand-measured surveyed clearances on a small gold set. If no cross-modality oracle is feasible on CPU, then H3a/H3b are NOT achievable in Stage 2; say so and ship H1 + the re-voxelization sensitivity as the honest Stage-2 result, rather than presenting a same-modality agreement as external validation.
- **blocker**: The escape hatch ('if Occ3D shares the pipeline, accumulate a DISJOINT sweep window') is logically self-defeating for STATIC geometry and renders H3a unfalsifiable. The entire scope is static free-space (a wall/curb that does not move). A disjoint temporal window of the SAME scene either (a) re-images the SAME static walls from the SAME LiDAR sensor -> errors still correlated, independence not achieved; or (b) the scene actually differs between windows -> the oracle now measures a DIFFERENT geometric quantity than the predicate, so disagreement no longer means predicate error and agreement is meaningless. Both horns kill the validity claim. The plan treats this as a checkbox ('confirm the two pipelines are not the same operation') but provides no operational test that could come out 'circular,' so the 'oracle still circular' kill can never actually fire.
  - fix: Replace the vague 'disjoint window' clause with a concrete, pre-registered independence test that CAN fail: e.g. compute the spatial error-correlation between predicate and oracle clearance on a held-out set against a third gold reference; require corr below a pre-set bound to call them independent. If you cannot specify a test whose failure is observable before reading H3 results, drop the independence claim entirely. Do not list 'oracle still circular' as a kill criterion unless there is a procedure that would trip it.
