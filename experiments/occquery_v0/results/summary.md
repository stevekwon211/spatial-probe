# occquery_v0 -- results summary

occquery is a deterministic instrument that **MEASURES box-blind static occupancy geometry**
(free-width, clearance, free-path) -- physical quantities a box-only query language cannot express.
It does NOT judge danger: whether a measured situation is dangerous (a lead car vs a wall, a near-miss
vs a parked car) is relative motion over time = the **dynfield** experiment (PLAN s4 Scope). occquery
is judged on MEASUREMENT accuracy + expressivity, never on retrieving "danger".

Result classes, kept distinct: unit/integration tests (`python -m pytest -q`); synthetic smoke
(`smoke.md`, generated by `run.py`); real-data measurement on Occ3D-nuScenes mini. An externally
validated denotation (measurement accuracy vs an *independent* oracle) is **NOT done** -- and on this
hardware it cannot be (see H3 below). The committed hypothesis + kill criteria are in
`../preregistration.md`; that file wins over the original `PLAN.md` for this run.

## H1 -- expressivity (oracle-free, the headline) -- HOLDS

Box-blind witness (`tests/test_expressivity.py`, 3 families: clearance / free-path / corridor): two
scenes with identical tracked-box + map + velocity observables differing only by an unboxed occupancy
obstacle -- any function of the box+map channel MUST return the same answer; the occupancy predicate
distinguishes them. This is non-identifiability under the box+map observable set, by construction, no
oracle. RefAV's released 32-function set has 0 free-space primitives (`docs/expressivity-vs-refav.md`).

The pre-registered query set is now **24 queries** (was 4), committed before any val run
(`queries.yaml`, 2026-06-22): **20 occupancy** (5 each across clearance / centerline / free-path /
corridor, satisfying the preregistration's "N ≥ 20 occupancy queries") + 4 box-baseline fairness
controls. Expressibility coverage on the set: **occupancy 24/24, RefAV 4/24** -- the 20 occupancy-native
queries are inexpressible in the box-only language, the 4 box controls are (and the box backend should
win them, by design). Thresholds are pre-registered knobs reported as curves (clearance 0.3/0.5/1.0 m
with and without a speed gate; corridor ==0 / <0.5 / <ego-width; free-path horizons 0.5/1/2/4 s), none
tuned to data.

## H3 -- denotation correctness -- DEMOTED to internal-consistency

Per `preregistration.md`, H3 is demoted until an independent oracle is earned. An independent oracle
needs a different MODALITY (stereo / MVS / ruler) AND a measured independence test; neither is feasible
here (no raw LiDAR sweeps on disk, no second sensor on a Mac). What exists today:

- **Scoring kernel** (`src/probe/metrics.py`): denotation P/R/F1 + cluster bootstrap N=1000 CIs + the
  relative-gap-vs-box-only curve. No absolute F1 gate is representable (the 0.90 cutoff is gone; the
  decision is read off a curve). Built + adversarially verified this session (3 independent skeptics,
  executed break-probes, SURVIVED 0/3; the rare/zero-positive paths report "undefined", never a
  vacuous F1=1.0). 17 unit tests.
- **Consistency probe** (`consistency.py`, run on all 404 mini frames). Re-derives each measurement by
  a second computation over the SAME Occ3D voxels and reports the residual -- a CONSISTENCY signal,
  same data source, **not** denotation correctness:
  - `min_free_width` (the one genuinely different family: raster index-width vs continuous
    nearest-neighbour width): |residual| median **0.0 m**, p95 = max = **0.40 m** (exactly one voxel),
    signed mean **+0.003 m**. So the 0.4 m BEV rasterisation shifts the corridor-width measurement by
    at most one voxel, unbiased. Bounds re-voxelisation rounding only.
  - `lateral_clearance` / `centerline_lateral_distance`: residual **identically 0** across all 404
    frames -- a **tautology**, because the predicate is ALREADY exact point-NN over the voxel centers
    (`clearance.py`), so a point-NN reconstruction is the same algorithm. Reported as a negative, NOT
    as agreement that validates anything.
- **The real denotation P/R/F1 is BLOCKED.** It needs (a) a positive-containing scene set -- the 10
  mini scenes have **0 positives** for the curated queries, so recall is unmeasurable; and (b) the
  Occ3D val occupancy labels, not on disk. The kernel + sealed queries are ready: the number is one
  data download + a hand-labeling/sealing step away (below).

## Unknown-policy sensitivity on real mini (the conditioning signal)

Each occupancy query was run under unknown=free vs unknown=occupied on the per-frame **masked** lidar
view (~78% of voxels unobserved). Nearly every occupancy query is **unknown-SENSITIVE**: unknown=occupied
(treat the unseen as solid) fires on ~all 10 scenes, unknown=free (treat the unseen as empty) on few
(e.g. `free_path_is_blocked`: free → {scene-0103}, occupied → all 10; `tight_clearance_at_speed`: free
→ {}, occupied → 6). The truth is bracketed between the two rules.

This is **expected, not a failure** -- it is the section-4 kill-shape the plan predicts on the masked
view (PLAN L173-174), and it is exactly the conditioning variable the gt-distrust / vis-calibration
experiments are built on. The H3 oracle is therefore defined on the **dense** GT (unknown ~0, denotation
stable -- verified 2026-06-20), not the masked view. One honest sub-finding: the corridor predicate
(`min_free_width`, C-space reachability + a min_cluster_voxels=2 noise gate) is MORE unknown-robust than
raw `lateral_clearance` -- 3 of the corridor queries are stable (empty) under both rules, only
scene-0061 fires for the ego-width corridors. The reachability + cluster filtering absorbs scattered
unobserved-as-occupied voxels that the raw nearest-obstacle clearance does not.

## The lead-car finding (why the v0 "0/5 precision" was a category error)

Walking scene-0061 in the 3D viewer showed its 0.80 m free-width is a CORRECT measurement of the gap
beside a **lead car** (the ego follows it, decelerating 5.1 → 2.2 m/s over 36 m) -- not a false
positive. A static measurement was being graded against a dynamic-danger answer key. The danger verdict
(lead car vs wall) belongs to dynfield. Full account: `docs/h3-real-data-findings.md`.

## What is blocked, and exactly how to unblock it

- **H3 real number (val denotation).** Download the **Occ3D-nuScenes val occupancy labels** (from the
  Occ3D release, NOT nuScenes.org; the nuScenes side -- `v1.0-trainval_meta` -- pairs with it and is
  already downloaded). Then: seal a positive-containing scene set in `held-out.txt` BEFORE scoring,
  hand-label query membership, and run the (already-built) kernel. The pre-registration discipline is
  the point: the queries and criteria are sealed now, the data opens once.
- **H2 leaderboard.** occquery's denotation arm is on Occ3D-**nuScenes**; the RefAV HOTA-Temporal anchor
  is on **Argoverse 2** -- a different dataset (`docs/m2-adapter-contract.md` calls this the single
  biggest open question). Recommended: keep the substrates separate (H1/H3 on Occ3D-nuScenes, H2 a
  competitiveness check of the retrieval engine on the box-expressible query subset on Argoverse 2).
  Sourcing occupancy for Argoverse 2 would mean generating a non-released oracle, which undermines the
  released-oracle property H3 wants. H2 is a credibility leg; it is NOT required for the H1 headline.
- **Independent oracle.** Not buildable on this Mac (same-sensor LiDAR cannot be its own independent
  modality). Deliberately left unbuilt rather than faked; the consistency probe above is the honest
  ceiling on the available data.

## Decision: PURSUE (H1 is the result; H3/H2 are expansion legs)

H1 (expressivity) is strong, oracle-free, and **done** -- a defensible contribution on its own. The
instrument MEASURES box-blind geometry and v1 made that measurement honest (C-space reachability,
noise-robust, rasterisation rounding bounded to one voxel). H3's number and H2's leaderboard are
expansion legs gated on a data download and a substrate decision respectively, both named above. Danger
/ dynamics is the next experiment (dynfield), not occquery.
